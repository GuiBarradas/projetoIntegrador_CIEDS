Treinamento
Etapa 1
Criar um banco de dados no SQL Server e fazer uma tabela com os seguintes campos: Código(PK - “chave primária"), nome, ativo, CPF, data de nascimento (Os campos não devem possuir acentos e nem espaçamentos no SQL Server, nem nas variáveis usadas no sistema) e fazer o protótipo na tela com os mesmos campos criados no SQL. Obs: CPF e data de nascimento devem ter suas respectivas máscaras de campos para facilitar o preenchimento e evitar erros. Iniciar
Etapa 2
Gravar os dados da 1º etapa no banco de dados diretamente pelo sistema e construir o filtro apresentando: Nome, data de nascimento, ativo e CPF. Recuperar todos os dados anteriormente gravados pelo filtro e excluir logicamente (Inativar o registro). No filtro deverá ser possível pesquisar por nome, ativo, CPF e data de nascimento. Iniciar
Etapa 3
Impedir que sejam gravados os seguintes campos sem o devido preenchimento: Nome, CPF e data de nascimento. Os campos devem ficar na cor amarela para indicar que são campos obrigatórios utilizando a classe "required"; Apresentar na tela a idade do funcionário de acordo com a data de nascimento. O campo idade deverá ficar na cor cinza indicando que está bloqueado / não deve ser preenchido usando a classe "readonly" e atributo "readonly"; Validar a data de nascimento do funcionário (Não permitindo informar uma data futura ou inexistente exemplo: "27/10/2077" ou "99/99/1234"); Aceitar somente CPF válidos e verificar se o CPF já foi cadastrado no sistema; Não permitir 2 funcionários com o mesmo CPF. Iniciar
Etapa 4
Incluir no cadastro do funcionário o RG (Com sua devida máscara) e o sexo cadastrado em uma tabela SEPARADA no banco de dados como FK (Chave Estrangeira) <- Pesquisar sobre;
Validar o RG para não permitir duplicidade, incluir o campo "Estado Civil" como uma combo (Semelhante ao ativo) e no filtro de cadastro de funcionários criar parâmetro de filtragem a partir de data de nascimento inicial e final. Iniciar
Etapa 5
Fazer um novo accordion (Divisão dentro da tela) para contato e dentro dele vão ter duas listas: Uma para telefone e outra para email. Deverá ser possível adicionar itens na lista, remover os itens selecionados e, ao clicar no hiperlinks da lista, os dados deverão voltar aos seus respectivos campos para que possíveis alterações possam ser realizadas; Quando gravar, a tela deverá salvar todos os itens na lista, se existirem, em uma tabela filha, referenciando a tabela pai e vincular-la por FK e, ao recuperar, todos os dados da lista devem ser preenchidos; Validar os itens em ambas as listas, não permitindo 2 telefones ou emails iguais, também não permitir 2 itens principais na lista. Ex: 2 telefones principais. Iniciar
Etapa 6
Criar um novo accordion de endereço contendo os seguintes campos: CEP, logradouro, número, complemento, UF, bairro, cidade e gravar e recuperar esses campos (Na mesma tabela). Implementar busca por CEP, preenchendo todos os dados do endereço de forma automática. Criar uma nova tela (Separada) de "Tipo de Dependente", com código, descrição e ativo com todas as funcionalidades de gravar, recuperar e excluir. Fazer o filtro para a tela de tipo dependente buscando pela descrição e por ativo. Criar campos: Primeiro emprego (Obrigatório) e PisPasep (Se for primeiro emprego. o candidato não precisa informar o pis e se tiver preenchido “SIM" o valor deverá ser apagado e bloqueado no readonly). Obs: Ao recuperar o cadastro, o sistema também deverá bloquear ou liberar os campos. Iniciar
Etapa 7
Montar um menu lateral por telas (No config.ui.php); Cada tela deverá pertencer ao seu menu e todos os itens da tela devem ser coerentes ao seu menu lateral; Na parte superior da página também precisará estar referenciado, de forma correta, de acordo com o menu. Adicionar um ponteiro no menu lateral mostrando ao usuário qual página ele atualmente está; Essas configurações precisam ser feitas em TODAS as telas. Iniciar
Etapa 8
Na tela de cadastro de funcionário criar um novo accordion para dependentes, contendo uma lista grande com os seguintes campos: Nome, CPF, data de nascimento e tipo dependente (Da tela cadastrada anteriormente(FK)); É necessário validar a lista, adicionar e realizar todos os processos feitos nas listas anteriores e também é preciso ver se o CPF do dependente é válido; Obrigar o usuário a registrar pelo menos uma forma de contato antes de gravar o cadastro; Nas tabelas básicas de Sexo e Tipo Dependente verificar se foi feita uma validação para impedir cadastrar dois nomes iguais para não haver duplicidade.
